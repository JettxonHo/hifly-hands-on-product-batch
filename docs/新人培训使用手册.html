先读这里

这份手册解决什么

项目通过飞影网页的“一键成片 - 手里有货”模式批量制作商品种草视频。该模式当前没有开放官方 API，因此使用浏览器自动化完成批量上传、生成和下载。

客户负责提供可用素材和商品信息；运营负责校验素材、维护人物图池、运行任务和质检成片。登录、验证码、页面改版和生成排队等外部状态，仍需要人工关注。

开始前检查

  1. 确认飞影账号可进入“手里有货”页面，并有足够积分。
  2. 确认客户素材齐全，商品图清晰且产品名称、卖点、品类已经确定。
  3. 先用 5 至 10 条商品做校准批次，再扩到 20 至 50 条。
  4. 不要把账号密码、登录态或真实客户素材提交到 Git 仓库。

关键限制

当前“手里有货”页面没有提供数字人姿势参数。数字人的坐姿、站姿、景别和场景主要由人物素材图或飞影推荐模板决定。只使用同一张人物图或同一推荐模板时，批量视频容易出现姿势和背景雷同。

给客户和合作方

客户协作区

客户只需按下面的规则提交素材。内部运营会在导入前再次校验，素材越完整，批量出片的稳定性和差异化越高。

客户需提交的内容

必需

  • 商品图片：每个商品至少 1 张。
  • 产品名称：建议 20 字以内，方便口播和字幕展示。
  • 核心卖点：建议不超过 3 条，用分号分隔。
  • 商品品类：如 fresh_food、beauty、snacks，用于匹配人物素材池。

建议提供

  • 人物/背景图：能显著降低批量视频的背景与姿势重复。
  • 品牌禁用词与合规要求：尤其是食品、功效、医疗、金融相关商品。
  • 期望风格：例如亲和日常、专业讲解、轻快种草。
  • 验收要求：成片比例、时长、字幕、交付数量与命名规则。

商品图片要求

项目要求原因
格式JPG 或 PNG飞影上传弹窗支持这两种格式。
大小单张不超过 10MB超过限制会直接上传失败。
主体商品完整、清晰、无遮挡降低“手持”合成时的形变和误识别。
画面优先白底图或干净场景图复杂背景会增加合成画面的不确定性。
文字包装文字可读，但不承诺逐字还原生成模型可能让小字产生变形。

人物与背景素材建议

若客户希望批量结果更像不同达人在不同环境下推荐商品，建议每个品类至少准备 3 至 5 张竖图，并刻意拉开差异：

  • 混合坐姿、站姿、半身和中景；不要全部用同一种姿势。
  • 混合厨房、客厅、梳妆台、办公桌等与品类相关的自然场景。
  • 人物手部和身体轮廓要清楚，避免原图就有严重遮挡。
  • 避免人物手中已有同类商品，以免生成图出现多个商品或手部冲突。

为什么背景会重复

这是素材与飞影模板共同造成的结果，并不是批量脚本故意固定背景。当前脚本会优先使用客户指定的人物图；没有时才从对应品类人物池轮换；再没有才使用飞影推荐人物。人物素材不足时，成片自然会趋同。

商品信息表字段说明

字段谁填写填写说明
sku运营唯一编号，例如 SKU001。
product_name客户产品全称或视频展示名。
selling_points客户核心卖点，用分号分隔。
category客户/运营对应人物素材池目录名。
image_path运营商品图相对路径，例如 products/images/SKU001.png。
person_image_path客户/运营可留空；留空时使用人物素材池轮换。
status运营新任务填写 pending。

客户验收清单

  • 视频文件可正常播放，且不是 0 字节文件。
  • 画面比例、时长、字幕和交付数量符合双方约定。
  • 商品外观无明显变形，包装信息没有严重错乱。
  • 数字人口型、动作和音频没有明显卡顿或不同步。
  • 产品名称、卖点、字幕和 CTA 没有错字、夸大或禁用表述。
  • 同一批视频在数字人姿势、背景和景别上没有过度重复。

给内部运营人员

运营执行区

请按顺序执行。跳过弹窗中的“确认”会导致下一条任务仍停留在上一条的结果弹窗内，是当前页面最容易被忽略的一步。

1. 首次环境准备

本机需要 Node.js 20+（推荐 22+）、npm 和 Playwright Chromium。进入项目根目录后依次执行：

npm install
npx playwright install chromium
cp config.example.json config.local.json

config.local.json 仅保存本地校准项，不能提交到 Git。

2. 登录飞影并保存登录态

首次使用或登录失效后执行：

npm run login
  1. 在弹出的浏览器中完成飞影登录。
  2. 确认可以进入 手里有货页面。
  3. 回到终端按 Enter，脚本保存当前浏览器登录态。

3. 准备商品信息与素材

  1. 把商品图放入 products/images/，建议按 SKU 命名。
  2. 填写 products/products.csv 或 products/商品信息表.xlsx。
  3. 为每条商品填写唯一 sku、产品名、卖点、category 和商品图路径。
  4. 新任务将 status 填为 pending。

4. 配置人物素材池

人物图的选择顺序固定如下：

  1. 商品表中明确填写的 person_image_path。
  2. 同品类目录 assets/person_pool/<category>/ 中的图片，脚本会逐条轮换。
  3. 默认目录 assets/person_pool/default/ 中的通用人物图。
  4. 以上都没有时，按本地配置退回飞影弹窗中的推荐人物。

人物池支持 JPG、JPEG、PNG。推荐将人物图按品类放置，例如：

assets/person_pool/fresh_food/
assets/person_pool/beauty/
assets/person_pool/snacks/
assets/person_pool/default/

5. 生成前校验

在消耗积分前必须运行：

npm run validate

校验会检查必填字段、商品图路径、显式人物图路径，以及无人物图时是否存在匹配的品类池、默认池或允许使用飞影推荐人物。

6. 开始批量生成

npm run run

脚本会逐条进入 手里有货 页面，并按照下方流程处理。建议首次校准批次保持 5 至 10 条，确认成片质量和下载规则后再扩大批量。

页面关键流程

  1. 在页面外层点击“上传人物+产品图”。
  2. 在弹窗内上传或选择人物图，并上传商品图。
  3. 点击弹窗内的“立即生成”。此步骤生成手持商品图，会消耗积分。
  4. 等待生成完成后，点击弹窗右下角的“确认”。
  5. 回到页面外层，点击紫色“立即生成”，提交视频制作。
  6. 任务完成后，在“最新作品”卡片右下角点击下载图标。

必须点击“确认”

弹窗中的“确认”不是可选动作。若生成完成后没有点击它，下一次点击“上传人物+产品图”时，会继续停留在上一条商品的完成弹窗，批处理无法进入下一条。只点击右上角关闭或直接重新上传都不应作为正常流程。

7. 下载、归档与质检

目录内容运营动作
downloads/下载的视频样片或成片检查文件能否播放并按项目规则归档。
logs/JSONL 运行日志和失败原因定位失败商品与页面状态。
screenshots/异常截图和调试截图复核页面报错、弹窗或下载状态。
outputs/最终交付打包目录在质检通过后输出给项目负责人。

遇到异常时按这里处理

常见问题排查

登录失效、要求验证码或无法进入手里有货页面

重新执行 npm run login，在浏览器中人工完成登录或验证码，然后按 Enter 保存登录态。确认浏览器可以访问 https://hifly.cc/goods 后再运行批次。

商品图或人物图上传失败

检查图片是否为 JPG/PNG、是否小于 10MB、路径是否正确、文件是否可打开。商品图需要主体清晰；人物图避免严重遮挡和手中已有同类商品。修正后运行 npm run validate。

页面没有找到“立即生成”按钮

先确认是否位于手里有货页面。外层和弹窗各有一个“立即生成”：弹窗内用于生成手持商品图，外层用于生成视频。如果飞影页面文案或结构更新，需要在 config.local.json 的 hiflyUi 配置中重新校准，不要直接盲点其他按钮。

手持商品图已生成，下一条任务却仍卡在弹窗

这是没有点击生成结果弹窗中“确认”的典型表现。点击右下角“确认”回到外层页面，再继续外层视频生成或下一条任务。不要把右上角关闭当作确认。

数字人姿势总是坐着，画面太单一

当前飞影手里有货页面没有暴露姿势控制，因此这不是脚本遗漏的参数。优先为同一品类准备不同站姿、坐姿、景别和场景的人物图，并让商品表留空人物图时从品类池轮换。只使用飞影推荐人物时，风格和姿势更容易重复。

背景过于雷同

增加 assets/person_pool/<category>/ 下的差异化人物/背景图，每个品类建议至少 3 至 5 张。客户若提供单张固定人物图，脚本会优先尊重该选择；要获得变化，需要客户提供多张不同场景素材或允许运营改用人物池。

视频生成失败、长时间排队或下载失败

先查看 logs/ 和 screenshots/，确认是否为积分、队列、网络或页面异常。降低单批量并重试；下载问题则回到“最新作品”确认任务已完成，再检查浏览器下载权限和 downloads/ 目录。失败商品修正后改回 pending 再运行。

交付前最后检查

  1. 运行 npm run validate，确认商品表和素材路径无误。
  2. 查看 downloads/，逐条抽检文件可播放性、商品质量、字幕和口型。
  3. 查看 logs/ 与 screenshots/，处理失败任务。
  4. 将通过质检的文件放入交付目录，并按客户命名规则打包。
