# 飞影批量生产本地 GUI 工作台设计

## 目标

在现有 Node.js + Playwright 自动化项目上增加一个 Mac 和 Windows 通用的本地网页工作台。使用者在自己的电脑上完成商品信息录入、素材上传、任务校验、积分确认、飞影批量生成、进度查看与成片下载，不再手工编辑 CSV 或执行多个终端命令。

使用者同时承担客户和运营角色。本期不建设公网服务、账号体系、多人协作、远程文件存储或跨设备任务同步。

## 运行方式

- 使用 Node.js 启动本地服务，并自动打开默认浏览器。
- 服务仅监听 `127.0.0.1`，不接受局域网或公网访问。
- Mac 和 Windows 使用同一套业务代码；启动脚本和路径处理必须跨平台。
- 飞影账号密码不进入工作台；自动化继续复用本机已有的 Playwright 登录态。
- 所有商品素材、任务数据、日志、截图和成片保存在本机项目目录。

## 系统结构

### 浏览器工作台

提供四个主要工作区：

1. 新建商品：单条填写产品名称、核心卖点、品类，上传商品图，可选上传人物图。
2. 批量导入：上传 Excel/CSV 和商品图片，解析后显示字段、图片匹配和重复 SKU 问题。
3. 待执行任务：预览、编辑、删除、批量选择、校验并启动生成。
4. 运行记录：查看逐条状态、失败原因、异常截图、重试操作和成片下载入口。

### 本地 Node.js 服务

负责：

- 接收和校验表单、图片、Excel 与 CSV。
- 为上传文件生成安全且不冲突的文件名。
- 按批次建立独立目录并持久化任务状态。
- 将界面数据转换为现有商品表字段。
- 调用现有校验模块和飞影批量执行器。
- 将运行日志、任务进度和输出文件映射回工作台。

### 飞影自动化执行器

继续复用现有 Playwright 主链路：

1. 进入 `https://hifly.cc/goods`。
2. 打开“上传人物+产品图”弹窗。
3. 上传或选择人物图，上传商品图。
4. 点击弹窗“立即生成”。
5. 手持商品图生成完成后点击“确认”。
6. 点击页面外层“立即生成”提交视频。
7. 等待最新作品并下载成片。

GUI 不复制这套页面逻辑，只通过稳定接口调用执行器并接收结构化状态。

### 公共模块接口

现有 `run-batch.js` 和 `validate-products.js` 必须拆为可复用业务模块与薄 CLI 适配层。GUI 不启动子进程、不解析控制台文本，也不调用会在模块加载时执行或 `process.exit()` 的文件。

业务模块提供以下稳定契约：

```js
validateProducts({ products, config, batchPaths })
// => { valid, items, errors, warnings }

runBatch({ batchId, items, config, paths, signal, onEvent })
// => { status, items, startedAt, finishedAt }
```

- `errors` 包含 `sku`、字段、错误码和可读信息；校验失败不抛出进程级退出。
- `onEvent(event)` 发出批次、商品、阶段、时间和证据路径；GUI 只消费结构化事件。
- `signal` 仅允许在安全检查点停止；已提交到飞影的任务不能伪装成未开始。
- CLI 的 `npm run validate` 和 `npm run run` 调用同一业务模块，继续保持现有使用方式。
- 配置根目录根据模块文件位置或显式 `projectRoot` 确定，不依赖启动时的 `process.cwd()`。

真实飞影与假执行器实现同一 adapter：

```js
createAsset(task, context)
submitVideo(task, asset, context)
querySubmission(remoteEvidence, context)
downloadArtifact(remoteEvidence, destination, context)
reconcileSubmission(task, checkpoint, context)
```

adapter 事件固定包含 `type`、`batchId`、`taskId`、`executionKey`、`phase`、`timestamp` 和可选 `evidence`。每个方法开始与完成时均发事件；`submitVideo` 的调用前检查点由服务持久化，返回后必须提供远端证据。假执行器按同样顺序发事件，并可配置在任一检查点失败或中断。

## 数据模型与目录

每个批次至少包含：

- `batch_id`：系统生成的唯一批次标识。
- `created_at`、`updated_at`：创建和更新时间。
- `status`：批次状态。
- `items`：商品任务数组。
- `estimated_points`：执行前展示的预计积分。

每条商品任务沿用现有字段：`sku`、`product_name`、`selling_points`、`category`、`image_path`、`person_image_path`、`script`、`avatar`、`voice`、`duration_seconds`、`status`、`retry_count`、`output_path`、`error_message`。另持久化 `task_id`、`resolved_person_image_path`、`execution_key`、`submitted_at`、`remote_evidence` 和 `artifacts`。人物图在校验通过、展示积分确认前冻结到 `resolved_person_image_path`；编辑任务、重新选择人物池或复制任务时重新冻结并使旧确认失效。

建议批次目录：

```text
workspace/batches/<batch_id>/
  batch.json
  products.csv
  uploads/products/
  uploads/people/
  downloads/
  logs/
  screenshots/
```

所有业务路径存储为项目内相对路径，运行时通过标准路径 API 转换为当前系统路径，不手工拼接 `/` 或 `\`。

## 操作流程

### 单条录入

1. 用户填写产品名称、核心卖点和品类。
2. 用户上传商品图，可选上传人物图。
3. 系统生成或校验 SKU，检查文件类型、大小和字段完整性。
4. 通过后加入待执行列表；失败时在字段旁显示可修正原因。

### 批量导入

1. 用户上传 Excel 或 CSV。
2. 用户一次选择对应的商品图片和可选人物图片。
3. 系统按确定性规则解析表格并匹配本次上传的图片。
4. 系统显示导入预览，标记缺失字段、重复 SKU、无匹配图片和无效文件。
5. 用户只提交校验通过的记录，或修正后重新校验。

导入规则固定如下：

- CSV 仅接受 UTF-8（可带 BOM）和逗号分隔；XLSX 默认读取第一个可见工作表，并在预览中显示工作表名称。
- 必填表头为 `sku`、`product_name`、`selling_points`、`category`、`image_path`；可选表头沿用现有商品表。表头去除首尾空格后精确匹配，不启用隐式中文别名；重复表头直接拒绝，未知列保留但不进入执行数据。
- 空白行忽略；公式单元格只读取工作簿中已保存的计算值，没有计算值时标错，不在服务端执行公式。
- SKU 作为字符串读取，去除首尾空格，保留前导零，按 Unicode NFC 和不区分大小写检测重复。
- 表格中的 `image_path` 和 `person_image_path` 只能写本次上传文件的原始文件名或逻辑名称，禁止绝对路径、父目录和任意本机路径。
- 匹配优先级为：表格精确指定的上传逻辑名称，其次是与 SKU 完全一致的文件名主体。比较时执行 Unicode NFC 和不区分大小写，扩展名仅允许 JPG、JPEG、PNG。
- 同一逻辑名称出现多次、同一 SKU 匹配多个扩展名或一个文件匹配多条任务时均标记为歧义，不自动猜测。

### 一键运行

数据流为：

```text
保存素材
→ 生成标准商品表
→ 执行现有校验
→ 显示任务数量与预计积分
→ 用户二次确认
→ 启动飞影自动化
→ 推送逐条进度
→ 关联下载成片
```

校验失败时禁止启动自动化。系统全局一次只允许一个飞影批次运行，因为所有批次共享同一浏览器登录态；其他批次排队或保持待执行。

## 状态与恢复

商品任务状态及允许操作：

| 状态 | 含义 | 允许操作 |
| --- | --- | --- |
| `needs_input` | 字段或素材待补充 | 编辑、删除、重新校验 |
| `validation_failed` | 校验未通过 | 编辑、删除、重新校验 |
| `pending` | 已校验、未确认执行 | 编辑、删除、选择执行 |
| `confirmed` | 快照和积分估算已确认 | 启动；任何编辑都会退回 `pending` |
| `generating_asset` | 正在生成手持商品图 | 等待、安全停止 |
| `asset_confirmed` | 已点击手持图“确认” | 继续提交视频 |
| `submitted` | 视频已提交到飞影 | 仅轮询远端，不允许重新生成 |
| `download_pending` | 远端视频完成、等待下载 | 仅重试下载 |
| `completed` | 成片已下载并校验 | 打开、归档 |
| `failed_pre_submit` | 提交飞影前失败 | 修正后可重新执行 |
| `failed_remote` | 飞影明确返回生成失败 | 人工确认后可重新生成 |
| `interrupted_unknown` | 提交边界不确定或进程中断 | 必须先核对远端作品，禁止直接重试 |

批次状态由商品状态汇总生成。服务在每个状态变化和飞影提交动作前后原子写入 `batch.json`。提交前先写入 `asset_confirmed` 和 `execution_key`，页面确认已提交后立即写入 `submitted` 与可获取的远端证据；下载完成后再写入 `completed`。

应用重新启动时扫描批次目录。原 `generating_asset` 和 `asset_confirmed` 任务变为 `interrupted_unknown`；`submitted` 和 `download_pending` 只恢复轮询或下载，不创建新视频。系统提供“打开飞影作品页并核对”动作，只有用户明确标记远端不存在时，任务才能转为 `failed_pre_submit` 后重新执行。

失败记录必须包含可读错误、错误码、阶段、时间、重试次数以及日志或截图 artifact ID。下载重试与视频重新生成是两个不同操作。

状态转移规则：

- `needs_input` 或 `validation_failed` 修正并校验通过后进入 `pending`。
- `pending` 完成快照与积分确认后进入 `confirmed`；编辑或重新选择任务会退回 `pending`。
- `confirmed → generating_asset → asset_confirmed → submitted → download_pending → completed` 是正常主链路。
- 提交前明确失败进入 `failed_pre_submit`；修正后必须回到 `pending` 并重新估算、确认和生成 `execution_key`。
- 飞影明确失败进入 `failed_remote`；任何重新生成同样必须回到 `pending` 并重新确认。
- 用户在安全检查点停止时，未提交任务回到 `pending`；验证码或登录暂停保持当前状态并增加 `paused_auth` 标记，不改变提交语义。
- 无法判断是否提交、无法唯一关联远端作品或异常退出时进入 `interrupted_unknown`，只能执行核对动作。
- 批次汇总优先级为：`interrupted_unknown` > 活跃执行状态 > `paused_auth` > 失败状态 > 待输入/待执行 > 全部完成。部分完成批次继续显示每条真实状态，不用单一批次状态覆盖商品状态。

### 远端作品关联与恢复

- 优先持久化飞影返回或页面可观察到的任务 ID、作品 ID 和作品 URL。
- 如果没有稳定 ID，提交前记录最新作品集合摘要，提交后记录提交时间、SKU、产品名、所选声音、时长和页面可观察标识。
- 恢复查询必须排除提交前已存在作品，并在时间窗口与任务标识上得到唯一匹配；匹配多个或没有足够证据时进入 `interrupted_unknown`。
- 不允许仅凭“最新作品”位置自动认领、下载或标记完成，因为期间可能存在人工创建或其他任务。
- `remote_evidence` 至少保存匹配方法、观察值、提交前集合摘要、首次/末次观察时间和证据截图 artifact ID。

### 全局执行锁与幂等

- `workspace/execution.lock` 是跨批次全局文件锁，使用排他创建原子获取，禁止“先检查、后写入”。锁包含服务实例 ID、PID、批次 ID、创建时间和心跳时间；内存锁仅作为同进程快速保护。
- 心跳默认每 5 秒刷新，超过 30 秒视为可疑而不是自动无效。活性判断同时核对实例 ID、PID 和心跳；PID 复用、实例不匹配或状态不确定时只允许人工接管。
- 获得锁后才可启动 Playwright。锁存在且对应进程/心跳有效时，其他运行请求返回冲突；陈旧锁不能自动触发重新生成，只能恢复为待核对状态。释放锁时必须核对实例 ID 和批次 ID，其他实例不能删除该锁。
- 每次确认生成 `execution_key`，绑定已选 `task_id`、所有执行字段摘要、人物图摘要、估算版本和确认时间。同一 key 的重复 HTTP 请求返回同一执行记录。
- 任务选择、内容、人物图、时长或配置变化会使确认和 `execution_key` 失效，必须重新校验和确认。
- 服务启动时若发现仍存活的旧执行器，不启动第二个执行器，并提示用户关闭旧进程或继续等待。

## 积分与高风险操作

- 估算配置包含版本化的 `assetPointsPerItem` 和可选 `videoPointsEstimate`。初始值由本地配置显式填写，不从历史截图硬编码；未知时显示“手持图/视频积分未知，请以飞影页面为准”，仍要求确认。
- “开始生成”前显示商品数量、单条手持图积分、视频积分估算和预计总积分；总计按已选任务逐项计算，未知项不以零代替。
- 由于飞影实际视频积分可能受页面配置和时长影响，预计值必须标注为估算，不能伪装为最终扣费。
- 二次确认弹窗列出批次名称、任务清单、人物图快照、任务数量、估算版本与预计积分。
- 用户确认后才启动 Playwright；关闭确认弹窗不产生任务。
- 运行中禁用重复启动按钮，并显示当前运行商品。

## 文件与本地安全

- 服务绑定 `127.0.0.1`，不使用 `0.0.0.0`。
- 只接受配置允许的 JPG、JPEG、PNG、CSV、XLSX 文件。
- 单张图片不超过 10MB；CSV/XLSX 不超过 20MB；单批不超过 500 个文件和 1GB。限制在读取或解压前执行。
- 不信任上传文件名；服务生成内部文件名，并使用原文件名仅作显示。
- 除扩展名外还要检查文件签名并实际解码图片；限制最大像素数。XLSX 解析器必须设置 ZIP 条目数和解压后总大小上限，禁止外部链接或宏执行。
- 上传先写入批次内临时文件，校验成功后原子改名；中断和过期临时文件在启动时清理。同名文件使用内部 ID，不覆盖已存在内容。
- 禁止目录上传和 symlink。所有解析后的目标路径使用 `realpath` 验证仍位于当前批次目录，防止目录穿越。
- 下载、日志、截图接口只接受 `batch_id + artifact_id`；服务端从 `batch.json` 清单解析真实路径，不接受客户端路径参数。
- 不通过 shell 拼接用户输入执行命令；业务模块以函数调用方式连接。
- 页面不读取、不展示、不导出浏览器登录态、账号密码或其他本地敏感数据。

### 本地 HTTP 请求真实性

- 服务启动时生成高熵随机会话令牌，只通过首次打开的本地页面引导和内存会话建立，不写入 URL、日志或持久化任务文件。
- 只接受预期的本地 `Host`；校验 `Origin` 与实际本地源一致，拒绝 `null`、跨域和 DNS rebinding 场景。
- 所有变更、确认、运行、重试和 artifact 请求必须携带自定义请求头及会话令牌；不接受可由跨站简单表单构造的变更请求。
- 设置严格 CORS 为不开放，使用 `SameSite=Strict` 会话 cookie，并验证内容类型。幂等 key 只防重复执行，不能代替请求授权。
- 页面响应设置限制性 CSP，禁止外部脚本、远端 iframe 和非本地连接。

## 界面原则

- 第一屏直接进入可操作工作台，不做营销落地页。
- 使用顶部批次信息和左侧/顶部视图导航，主区域展示紧凑表格与编辑面板。
- “新建商品”和“批量导入”使用清晰命令按钮；图片使用缩略图预览。
- 状态使用文本加图标，不仅依赖颜色。
- “开始生成”是唯一高风险主操作，使用二次确认；编辑、删除、重试等操作与其视觉层级分开。
- 窄屏保持可查看与简单编辑，但正式批量运行主要面向桌面浏览器。

## 错误处理

- 登录失效或验证码：暂停当前任务，提示用户在自动化浏览器完成处理后继续。
- 飞影按钮或结构变化：停止批次，保存日志与截图，不盲目点击相似按钮。
- 上传或表格解析失败：不写入待执行任务，返回到具体文件或行的错误。
- 生成排队或超时：保留任务状态和当前证据，允许人工检查后重试。
- 下载失败：保留飞影任务完成信息，允许仅重试下载，避免重复生成。
- 本地服务重启：恢复批次和非运行状态；中断任务需人工确认后才能重试。

## 跨平台运行约束

- 最低环境为 Node.js 20、macOS 12+、Windows 10+；首次安装执行 `npm install` 和 `npx playwright install chromium`。
- 统一启动命令为 `npm run gui`，不依赖 Bash、PowerShell 特有语法。无法自动打开浏览器时，终端明确输出本地 URL。
- 默认端口由配置指定；端口被占用时选择下一个可用端口并输出实际 URL，不绑定公网地址。
- 项目根目录从 `import.meta.url` 推导或由显式参数传入，不依赖当前 Shell 目录。
- 内部文件名不使用 Windows 保留名和尾随点/空格，并限制路径长度；业务路径始终通过 `node:path` 处理。
- 收到退出信号时停止接收新任务、等待安全检查点、刷新 `batch.json` 并释放全局锁。

## 验收标准

验收与测试阶段必须启动独立 subagent。该 subagent 不参与本功能实现，从验收者视角读取本规格、运行自动化测试、检查桌面与窄屏界面、审查安全边界和现有飞影流程回归，并输出带证据的通过/失败清单。主代理负责复核其证据、修复问题，并在修复后再次安排独立验收；不得仅以实现者自测作为完成依据。

测试必须使用可注入的假执行器，默认不访问飞影、不消耗积分。每个验收用例记录输入、操作、预期 HTTP/状态/文件结果和证据路径。最低矩阵包括：

- 单条创建、编辑、删除和字段级错误。
- UTF-8 CSV、XLSX 首个可见工作表、重复 SKU、重复表头、公式无缓存值、歧义图片匹配。
- 绝对路径、父目录、symlink、伪造 MIME、超大图片像素、XLSX 解压膨胀和 artifact 越权读取。
- 非法 Host/Origin、缺少会话令牌、跨站简单请求和 DNS rebinding 模拟请求。
- 未确认启动、确认后编辑、相同幂等 key 重复请求、不同批次并发启动和陈旧锁恢复。
- 在提交前、提交确认边界和下载前模拟崩溃，验证不会自动重复生成；验证下载专用重试。
- 使用假事件流验证任务状态转移、批次汇总、日志/截图/成片关联和重启恢复。
- 使用无远端 ID、重复候选作品和人工插入新作品的假数据验证唯一关联；歧义时必须进入 `interrupted_unknown`。
- 桌面主视口与 390px 窄屏的可用性、无重叠和关键按钮状态。
- macOS 与 Windows CI 运行单元/集成测试；两个平台至少各进行一次真实启动冒烟。飞影真实回归只使用明确授权的小批次。

- Mac 和 Windows 均可通过 `npm run gui` 打开本地工作台；自动打开失败时终端给出可访问 URL。
- 单条表单可创建、编辑、删除和校验任务。
- CSV/XLSX 可导入并显示字段、重复 SKU 与图片匹配错误。
- 上传文件类型、大小、文件名与目录穿越得到拦截。
- 工作台可生成与现有校验器兼容的商品表。
- 未通过校验或未确认积分时不会启动飞影自动化。
- 同一批次及不同批次都无法并发争用飞影登录态；重复幂等请求不会创建第二次执行。
- 运行进度、失败原因、日志、截图和成片可在界面关联查看。
- 重启本地服务后，已保存批次仍可恢复。
- 在提交边界崩溃后任务进入待核对状态，不会自动重新生成；下载失败只重试下载。
- 现有“确认手持商品图后再生成视频”的飞影流程保持通过。

## 非目标

- 公网访问、云部署、远程上传或移动端 App。
- 多用户账号、权限、审批或组织协作。
- 自动支付、充值或承诺飞影最终扣费金额。
- 绕过验证码、登录安全机制或飞影平台限制。
- 第一版自动更新、安装包签名或 Electron/Tauri 桌面封装。
