# GUI 人物与文案策略优化设计

## 目标

在已跑通的飞影「手里有货」批量链路上，增加可控的人物来源和文案来源策略，解决两个生产痛点：

- 批量视频数字人过于单一，容易出现同一人物、同一姿势、同一背景反复出现。
- 客户或运营有指定口播、合规话术、品牌表达时，不能完全依赖飞影 AI 自动文案。

本次优化不改变“一个商品生成一条视频”的 V1 口径，不引入公网服务、账号系统、多人协作或复杂素材库后台。

## 设计原则

- 保留当前已验证链路为默认路径：不填人物、不填文案时，继续使用人物池/飞影推荐和飞影 AI 自动文案。
- 策略必须显性化：GUI 中能看出当前批次使用什么人物策略和文案策略。
- 失败要停在提交前：人物图、商品图或自定义文案无法可靠应用时，任务进入 `failed_pre_submit`，不得继续消耗飞影积分。
- 先做可控小闭环，不做大而全素材管理系统。

## 人物策略

GUI 增加“人物来源”设置，批次保存为 `person_strategy`。

| 策略 | 含义 | 第一版行为 |
| --- | --- | --- |
| `auto_pool` | 自动匹配 | 默认。按商品 `category` 从本地人物池轮换；品类池为空时读 default 池；仍为空时使用飞影推荐人物。 |
| `fixed_upload` | 固定人物 | 整个批次使用用户上传的一张人物图。适合品牌指定达人或先做单风格样片。 |
| `hifly_recommended` | 飞影推荐 | 不上传人物图，直接选择飞影弹窗里的推荐人物。适合没有本地人物素材时快速跑量。 |

现有 `person_image_path` 字段继续保留最高优先级：如果某条商品显式指定人物图，该条商品使用指定图，不被批次策略覆盖。

### 人物素材建议

为了降低重复感，运营应优先按品类准备人物池：

```text
assets/person_pool/toy/
assets/person_pool/electronics/
assets/person_pool/beauty/
assets/person_pool/fresh_food/
assets/person_pool/default/
```

每个常用品类建议 3-5 张人物图，并刻意拉开差异：

- 坐姿 / 站姿
- 近景 / 半身 / 全身
- 居家 / 办公 / 户外 / 货架场景
- 不同年龄、风格、服装

## 文案策略

GUI 增加“文案来源”设置，批次保存为 `script_strategy`。

| 策略 | 含义 | 第一版行为 |
| --- | --- | --- |
| `hifly_ai` | 飞影 AI 自动文案 | 默认。保持飞影页面的“AI 自动生成”开启，不主动填口播文案。 |
| `provided_script` | 使用导入文案 | 每条商品必须有 `script`。执行前关闭飞影“AI 自动生成”，填入文案；缺文案则校验失败。 |
| `mixed` | 混合模式 | 推荐。商品有 `script` 时使用导入文案；没有 `script` 时使用飞影 AI 自动文案。 |

单条录入、批量录入和批量导入都支持 `script` 字段。批量表继续沿用现有列名 `script`，不新增中文别名。

### 飞影页面约束

当前代码已经会在 `product.script` 存在时尝试填入 `hiflyUi.scriptLabel` 对应字段，但这还不够。因为飞影页面默认打开“AI 自动生成”时，文案输入框可能不可编辑，或填入后仍被 AI 文案覆盖。

因此第一版必须补充自动化能力：

1. 当任务决定使用自定义文案时，定位并关闭“AI 自动生成”开关。
2. 确认文案输入区域可编辑。
3. 填入 `script` 并截图记录。
4. 如果找不到开关、输入框不可编辑、或填入后无法确认文本存在，则抛出提交前错误。

如果任务决定使用飞影 AI，则不填 `script`，保持 AI 自动生成开启。

## GUI 交互

### 单条录入

新增字段：

- 人物来源：默认 `自动匹配`
- 文案来源：默认 `混合模式`
- 口播文案：可选 textarea

当文案来源为 `使用导入文案` 时，口播文案必填；当为 `混合模式` 时可为空。

### 批量录入

每行新增 `script` textarea 或可展开输入区。批量顶部提供默认策略：

- 人物来源应用到整个批次。
- 文案来源应用到整个批次。

第一版不做每行单独切换策略，避免界面复杂。

### 批量导入

继续支持 CSV/XLSX 的 `script` 列。导入预览中显示：

- 是否包含自定义文案
- 当前文案策略下是否缺失必填文案
- 当前人物策略下是否能解析人物来源

## 数据模型

批次增加：

```json
{
  "person_strategy": "auto_pool",
  "script_strategy": "mixed",
  "fixed_person_image_artifact_id": null
}
```

商品任务继续使用现有字段：

```json
{
  "person_image_path": "",
  "resolved_person_image_path": "",
  "script": ""
}
```

执行前冻结实际人物结果到 `resolved_person_image_path`。如果最终使用飞影推荐人物，该字段为空，并记录 `resolved_person_source: "hifly_recommended"`。

## 自动化流程变更

### 人物决策

执行准备阶段按顺序解析人物：

1. 商品显式 `person_image_path`
2. 批次 `fixed_upload`
3. 批次 `auto_pool` 的品类池 / default 池
4. 飞影推荐人物

自动化执行时：

- 有 `resolved_person_image_path`：上传该人物图。
- 无 `resolved_person_image_path` 且允许飞影推荐：调用 `selectRecommendedPerson()`。
- 两者都没有：校验失败，不进入飞影页面。

### 文案决策

每条商品在执行前计算 `resolved_script_mode`：

- `hifly_ai`：不填文案。
- `provided_script`：必须填 `script`。
- `mixed`：有 `script` 用自定义文案；没有则用飞影 AI。

自动化执行时：

- `resolved_script_mode = "custom"`：关闭 AI 自动生成并填入 `script`。
- `resolved_script_mode = "hifly_ai"`：保持当前默认行为。

## 错误处理

- 自定义文案策略下缺少 `script`：校验失败，显示在 GUI。
- 自定义文案无法关闭 AI 自动生成：`failed_pre_submit`，不点击外层“立即生成”。
- 自定义文案输入框不可编辑或填入失败：`failed_pre_submit`。
- 人物池没有可用图，且策略不允许飞影推荐：校验失败。
- 固定人物上传文件非法：导入或保存阶段失败。

## 测试计划

无积分测试：

- 人物策略解析单元测试：显式人物、固定人物、品类池、default、飞影推荐兜底。
- 文案策略解析单元测试：三种策略、缺文案校验、混合模式分流。
- GUI API 测试：单条、批量录入、导入表保存策略字段和 `script` 字段。
- 自动化 adapter 测试：自定义文案会先关闭 AI 开关再填文本；失败停在提交前。

真实飞影验证：

1. 先跑 1 条自定义文案商品，确认页面文案可控。
2. 再跑 3 条小批量：一条有自定义文案，两条走飞影 AI；人物分别覆盖本地池和飞影推荐。
3. 验证下载文件、截图、批次状态和错误恢复。

真实验证前必须再次获得用户确认，因为会消耗飞影积分。

## 不做范围

- 不做云端素材库。
- 不做多人协作。
- 不做每条商品多人物/多脚本 A/B 生成。
- 不做复杂的数字人质量评分。
- 不爬取或反向调用飞影未公开接口。

## 开放问题

- 飞影“AI 自动生成”开关的 DOM 结构需要实机校准。若无法稳定控制，第一版自定义文案功能应默认标记为实验能力，并在 GUI 中提示“需要先跑 1 条校准样片”。
- 飞影推荐人物数量和排序可能由账号或页面状态决定。第一版只保证能选择推荐人物，不承诺按品类精准选择某一个推荐人物。
