# 飞影「手里有货」批量生产 SOP

## 0. V1 项目口径

当前项目只做 **每个商品 1 条标准视频**。多达人、多脚本、多卖点版本、多平台投放 A/B 测试暂不进入 V1，后续作为升级项处理。

当前推荐生产路径：

- 批量生产默认使用 GUI 中的“批量生成”后端，目前稳定路径是 Playwright 浏览器自动化。
- 勾选“同时录制抓包产物（不切换批量后端）”只会在 Playwright 运行时额外保存 HAR，用于后续抽取、脱敏、离线回放和单条 HTTP 联调；它不会把批量生产切换成抓包 HTTP。
- 抓包 HTTP 已跑通单条真实出片，但仍只作为单条授权联调入口，不建议直接用于多商品批量生产。
- 所有会真实访问飞影并可能消耗积分的操作，必须先由运营确认。

标准视频定义：

- 画幅：竖屏 9:16。
- 时长：15-25 秒，默认 20 秒。
- 结构：左侧数字人口播，右侧手持商品或商品近景，中间保留清晰分隔。
- 输入：商品图片、产品名称、核心卖点。
- 输出：每个 SKU 交付 1 条成片、1 条口播脚本、1 组飞影生成提示词。
- 命名：`sku_product_name_standard.mp4`，例如 `SKU001_山野小青菜_standard.mp4`。

## 1. 输入标准

客户至少提供：

- 商品图片：主体清晰、无严重遮挡，优先白底图或干净场景图。
- 产品名称：不超过 20 个字更适合口播和字幕。
- 核心卖点：建议 3 条以内，用分号分隔。
- 商品品类：用于匹配人物素材池，例如 `fresh_food`、`beauty`、`snacks`。

运营补充：

- `sku`：唯一编号。
- `person_image_path`：可为空。为空时脚本按 `category` 从人物素材池自动轮换。
- `script`：可为空；为空时后续由文案生成模块补齐。
- `avatar`、`voice`：为空时使用配置默认值。
- `duration_seconds`：建议 15-25 秒。
- `status`：新任务填 `pending`。

## 人物与文案策略

默认使用 `auto_pool + mixed`：人物自动匹配人物池，文案采用混合模式。

- 人物池按商品 `category` 轮换；对应品类没有可用人物时回退到 `default`，仍找不到时使用飞影推荐人物。
- `script` 留空时，混合模式继续使用飞影 AI 自动生成文案。
- 商品表或 GUI 中填写 `script` 时，混合模式会尝试关闭飞影页面的“AI 自动生成”开关并填入自定义口播。
- 真实飞影执行中，如果自定义口播无法可靠关闭 AI 开关或确认填入，任务会停在提交前，不会继续外层视频提交。

## 2. 单商品标准视频结构

每条视频采用固定 5 段式：

| 时间 | 内容 | 画面重点 |
| --- | --- | --- |
| 0-3 秒 | 开场钩子 | 数字人直接给出场景痛点或强利益点，商品在右侧第一时间露出。 |
| 3-7 秒 | 产品介绍 | 说清产品名称和适用人群，避免过度铺垫。 |
| 7-14 秒 | 核心卖点 | 按客户提供卖点讲 2-3 个利益点，右侧保持商品手持近景。 |
| 14-18 秒 | 使用感/信任感 | 口播转成体验化表达，例如“家里常备”“通勤也方便”。 |
| 18-25 秒 | 轻 CTA | 引导收藏、下单或进店，不做夸大承诺。 |

口播脚本控制在 80-120 个中文字符。字幕按短句切分，每屏不超过两行。

## 3. 批量生产流程

如果使用本地 GUI，推荐流程如下：

1. 运行 `npm run gui` 打开本地工作台。
2. 在“单条录入”“批量录入”或“批量导入”中提交商品。
3. 检查“待执行任务”中的批次详情，确认商品数量、人物策略、文案策略。
4. 需要保留抓包材料时，创建批次前勾选“同时录制抓包产物（不切换批量后端）”。
5. 点击“开始生成”，在确认弹窗中再次核对“一商品一条片”和预计积分说明。
6. 生成完成后，在“运行记录”或“批次详情”中查看输出文件；GUI 支持下载已登记产物，也支持复制输出路径。
7. 如果批次失败，优先在 GUI 中使用“重试失败批次”或“重新生成异常批次”，不要重新逐条录入。

CLI/文件流程如下：

1. 把商品图片放入 `products/images/`，按 SKU 命名。
2. 按品类把人物/背景图放入 `assets/person_pool/<category>/`。如果客户没有提供人物图，至少准备 `assets/person_pool/default/` 的通用人物图。
3. 在 `products/products.csv` 填入商品信息。
4. 运行 `npm run validate` 检查字段和图片路径。
5. 运行 `npm run prepare-standard` 生成口播脚本、飞影提示词和 `qc_report.csv`。
6. 人工抽查 `outputs/standard-video-assets/scripts/` 和 `outputs/standard-video-assets/prompts/`。
7. 首次使用运行 `npm run login`，人工登录飞影工作台。
8. 运行 `npm run run` 批量进入 `https://hifly.cc/goods` 的「手里有货」模式生成视频。
9. 检查 `downloads/`、`logs/`、`screenshots/`。
10. 失败任务修正后改回 `pending` 或保留失败状态由脚本重试。

当前飞影页面必须经过弹窗流程：

1. 外层点击「上传人物+产品图」。
2. 弹窗内上传人物图和商品图；如果商品表没有 `person_image_path`，脚本会按 `category` 从人物素材池轮换，素材池为空才选择推荐人物图。
3. 弹窗内点击「立即生成」。
4. 生成完成后点击「确认」。
5. 回到外层点击「立即生成」生成视频。
6. 在「最新作品」卡片右下角点击下载图标。

## 4. 批量节奏

V1 按三段推进：

1. 样片校准：先做 3 个不同品类商品，确认人物、场景、分屏、商品还原和下载链路稳定。
2. 小批量试产：每批 10-20 个 SKU，记录失败原因和返修率。
3. 稳定批量：每批 20-50 个 SKU，批次完成后统一质检和打包。

同一批次里不要临时混入新模板、新达人或新脚本结构。需要换风格时，另开批次。

## 5. 文案标准

脚本必须围绕客户提供的产品名称和核心卖点，不额外编造功效、认证、销量、价格或医学效果。

推荐句式：

- 开场钩子：`最近[场景痛点]的人，可以看看这个[产品名]。`
- 卖点展开：`它的[卖点1]比较适合[使用场景]，[卖点2]用起来更省心。`
- 体验表达：`日常[使用方式]很顺手，放在[场景]也方便。`
- 轻 CTA：`想要省心一点的，可以先收藏看看。`

禁用倾向：

- 绝对化：`第一`、`最强`、`永久`、`100%`。
- 无依据功效：医疗、减肥、祛病、治疗类表达。
- 虚假背书：未提供证明的明星、机构、销量、专利、认证。

## 6. 质检标准

每条视频交付前检查：

- 文件可播放，非 0 字节。
- 竖屏比例，时长符合客户要求。
- 商品外观没有明显变形，包装文字没有严重错乱。
- 数字人动作和口型自然，无明显卡顿。
- 卖点表达不夸大，不触发平台违禁词。
- 字幕、产品名、CTA 没有错字。
- 同一批次人物姿势、背景、景别不过度重复。

## 7. 交付标准

每批交付目录建议：

```text
outputs/standard-video-assets/
  videos/
    SKU001_山野小青菜_standard.mp4
  scripts/
    SKU001_山野小青菜_script.txt
  prompts/
    SKU001_山野小青菜_prompt.txt
  qc/
    qc_report.csv
```

`qc_report.csv` 至少记录：`sku`、`product_name`、`video_path`、`duration`、`product_ok`、`script_ok`、`subtitle_ok`、`final_status`、`notes`。

## 8. 风险与兜底

- 登录失效或验证码：人工登录后重跑。
- 页面按钮或输入框文案变化：更新 `config.local.json` 的 `hiflyUi` 文案。
- 上传失败：检查商品图片格式、大小和路径。
- 生成失败：记录错误截图，降低批量并重试。
- 成片质量差：更换商品图或降低对“真实手持”的承诺，进入人工复核。
- 姿势单一：当前飞影页面没有姿势参数，优先补充不同姿势/场景的人物素材池；只用飞影推荐人物时，批量结果容易雷同。
